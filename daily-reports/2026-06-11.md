## 구현 내용

**백엔드 변경 (backend/main.py)**

- `session_store` 딕셔너리로 세션별 대화 이력 서버 메모리에 저장
- `SYSTEM_PROMPT` 추가 — Groq API 호출 시 messages 배열 맨 앞에 삽입
- `ChatRequest`에 `session_id` 필드 추가 (기본값 "default"로 하위 호환 유지)

**프론트엔드 변경 (frontend/src/App.js)**

- `useRef`로 탭 세션 동안 유지되는 session_id 생성
- New Chat 버튼 클릭 시 화면 초기화 + 새 session_id 발급
- 작동하지 않는 UI 요소 제거: Recent Chats 버튼, 더미 대화 목록, 검색창
- 현재 작동하는 기능만 유지: New Chat, 사이드바 토글

**응답 시간 표시**

- 백엔드: LLM 호출 전후 time.time()으로 측정 후 elapsed 필드로 반환
- 프론트: 봇 말풍선 하단에 🕒 X.XX초 형태로 표시

**스트리밍 응답 (backend/main.py + frontend/src/App.js)**

- 백엔드: Groq API stream=True로 호출 → SSE(Server-Sent Events) 방식으로 토큰 단위 전송
- 프론트: ReadableStream으로 토큰 수신 → 봇 말풍선 실시간 업데이트
- 응답 시간은 스트리밍 완료 후 표시

**구현한 기능의 의미 (뼈대)**

- 대화 맥락 유지(session_store): 추후 대화 이력 관리 고도화의 기반
- 시스템 프롬프트: 추후 챗봇 도메인에 맞는 프롬프트 템플릿으로 고도화 예정
- New Chat(session 격리): 추후 사용자별 대화 저장 기능의 기반

## 방향성

**generator 구조 고도화 (본 프로젝트)**

- 현재: Groq API 직접 호출 (하드코딩)
- 목표: 팀장님(PM) 프로토타입 기반 모듈화 구조로 전환
    - LLM 교체 가능한 구조 (Groq → 팀 파인튜닝 모델로 스왑)
    - 프롬프트 템플릿 시스템 (챗봇 도메인에 따라 교체 가능)
    - RAG 워크플로우 연동 (벡터DB 완성 후)
    - 대화 이력 요약 (단순 잘라내기 → LLM 요약으로 고도화)
- 구현 방식: 모델팀이 LangChain으로 구성한 RAG 파이프라인을 백엔드에서 연동 예정

**RAG 연동 조건**

- 프롬프트 템플릿: 챗봇 종류(도메인)에 따라 내용이 달라짐 → 팀 협의 후 확정
- 벡터DB: 모델팀이 FAISS/Chroma 중 선택 후 확정 → 그 이후 연동 가능
- 연동 포인트: 백엔드 get_ai_response() 함수 하나만 수정

**미구현 항목 (본 프로젝트에서 진행)**

- 로그인 / JWT 인증
- 사용자별 대화 저장 (PostgreSQL 등 별도 DB 필요)
- DB 구축 → 로그인 → 대화 저장 순서로 진행 예정

**앞으로 구현할 기능**

- 자동 스크롤, 입력창 비활성화, 에러 메시지 고도화, 메시지 복사 버튼 (주제 확정 전 가능)
- RAG 연동, 파인튜닝 모델 교체 (주제 확정 후)
- 다크모드, 마크다운 렌더링, 모바일 반응형 등 프론트 개선

**대화 내역 저장 방식**

- 대화 이력 제한 슬라이더는 구현 완료, DB 연동 후 실질적 활용 가능
- 추후 개선: 단순 잘라내기 → LLM 요약 방식으로 고도화 예정
