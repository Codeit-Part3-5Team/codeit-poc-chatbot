## 1. 프로젝트 개요 (Project Overview)

### 1.1 프로젝트 배경
- 가상의 B2G 입찰지원 컨설팅 스타트업 **'입찰메이트'** 사내 RAG 시스템 구축 프로젝트.
- 하루 수백 건의 RFP(제안요청서)가 나라장터 등을 통해 발주되며, 한 건당 수십~수백 페이지에 달함.
- 컨설턴트가 모든 문서를 일일이 읽지 않고도 **주요 요구조건, 발주기관, 예산, 제출방식** 등을 빠르게 파악할 수 있도록 지원.

### 1.2 데이터 처리 팀의 미션
> **"RAG 시스템의 근간이 되는 고품질 데이터 파이프라인을 구축하고, 성능 평가의 기준이 될 골든 데이터셋을 확보한다."**

데이터 처리 팀은 RAG 시스템의 **입력 품질(Garbage In, Garbage Out)**을 결정하는 가장 중요한 역할을 담당. Retrieval/Generation 팀의 성과는 데이터 처리 팀의 결과물에 직결됨.

### 1.3 최종 목표 (Definition of Done)
| # | 산출물 | 완료 기준 |
|---|--------|----------|
| 1 | RFP 문서 로더 | HWP/PDF 100건을 안정적으로 텍스트 추출 (성공률 ≥ 95%) |
| 2 | 메타데이터 정제 파이프라인 | `data_list.csv` 100건 정규화 + 벡터DB 메타필드 매핑 완료 |
| 3 | 청킹 전략 v1.0 | 베이스라인(고정크기) + 심화(섹션기반) 2가지 구현 및 비교 |
| 4 | 민감정보 마스킹 모듈 | 개인정보(이름/전화/이메일/주민번호) 마스킹 규칙 적용 |
| 5 | 골든 데이터셋 v1.0 | 검증된 Q&A 쌍 **최소 60건** (단일/다중/후속/거부 4유형) |
| 6 | 데이터 처리 문서화 | 코드 주석 + README + 처리 결과 통계 리포트 |

## 2. 범위 정의 (Scope)

### 2.1 ✅ 포함 범위 (In-Scope)
- **문서 로딩**: `.hwp`, `.pdf` 2가지 포맷 텍스트 추출
- **메타데이터 처리**: `data_list.csv`의 컬럼 정규화 및 필터링용 필드 구축 (발주기관, 사업명, 예산, 공고일 등)
- **텍스트 전처리**: 공백/특수문자 정리, 페이지 헤더/푸터 제거, 표/목차 처리
- **청킹 전략**:
  - 베이스라인: 고정 크기(Recursive Character Text Splitter, chunk_size=1000, overlap=200)
  - 심화: 제안서 섹션 구조(예: "1. 사업개요", "2. 사업범위") 기반 의미 단위 청킹
- **민감정보 마스킹**: 정규표현식 기반 (휴대전화, 이메일, 사업자번호, 주민번호 등)
- **골든 데이터셋 구축**:
  - 단일문서 질의 (20건)
  - 다중문서 비교/종합 질의 (15건)
  - 후속(맥락 유지) 질의 (15건)
  - 답변 거부(문서 외 정보) 질의 (10건)
- **유사검색용 메타데이터 정규화** (오타/약칭 대응: "국민연금공단" ↔ "국민연금")

### 2.2 ❌ 미포함 범위 (Out-of-Scope) — 백로그로 이관
- 한글/워드 외 포맷(JPG 스캔본, XLSX 등) 처리
- OCR 기반 이미지 문서 처리
- 다국어(영문) RFP 처리
- 실시간 문서 수집 크롤러 구축
- 골든 데이터셋 100건 이상 확장 (이번 스프린트는 60건 목표)
- 자동화된 데이터 품질 모니터링 대시보드

> **⚠️ Scope Creep 방지 원칙**: "이것도 있으면 좋겠다"는 의견은 무조건 백로그(GitHub Issue)에 등록. 본 스프린트 내 추가 작업은 PM 승인 필수.

## 3. 방법론 (Methodology)

### 3.1 개발 방법론: 애자일 + 일단위 스프린트

## 7. 품질 보증 (Quality Assurance)

### 7.1 데이터 품질 기준
| 항목 | 목표치 | 측정 방법 |
|------|--------|----------|
| 문서 로딩 성공률 | ≥ 95% (100건 중 95건 이상) | 처리 로그 카운트 |
| 메타데이터 결측률 | 핵심 필드(사업명, 발주기관) 결측 0% | `pandas.isnull()` 통계 |
| 청크 길이 분포 | 평균 800~1200자, 표준편차 < 500 | 히스토그램 분석 |
| 민감정보 마스킹 정확도 | 샘플 50건 검수 시 누락 ≤ 2건 | 수동 검수 |
| 골든셋 검증율 | 100% 상호 검증 완료 | 검증 로그 |

### 7.2 코드 품질 기준
- **함수 단위 docstring** 필수 (입력/출력/예시)
- **타입 힌트** 권장 (`def load_hwp(path: str) -> str:`)
- **에러 핸들링**: 파일별 try-except로 한 건 실패가 전체 중단으로 이어지지 않도록
- **로깅**: `logging` 모듈 사용, 처리 결과/실패 사유 모두 기록

### 7.3 골든 데이터셋 작성 가이드라인

> 💡 **골든 데이터셋이란?** AI 모델의 성능을 평가하고 검증하기 위해 사람이 직접 검증하여 신뢰성을 확보한 고품질의 표준(Reference) Q&A 데이터셋.

**작성 원칙**:
1. **답이 문서에 명확히 존재**해야 함 (출처 페이지/섹션 명시)
2. **모호한 질문 지양** ("좋은 사업인가?" ❌ → "예산이 5억 이상인가?" ✅)
3. **4가지 유형 균형** 필수:

| 유형 | 비중 | 예시 |
|------|------|------|
| 단일문서 추출 | 20건 | "국민연금공단 이러닝시스템 사업의 예산은?" |
| 다중문서 종합 | 15건 | "교육 관련 사업을 발주한 모든 기관을 알려줘" |
| 후속(맥락) 질의 | 15건 | (선행 Q 이후) "그럼 모니터링 요구사항은 있나?" |
| 답변 거부 | 10건 | "이 사업의 최종 낙찰업체는 누구인가?" (문서 외) |

**JSON 스키마**:
```json
{
  "id": "Q001",
  "type": "single",
  "question": "국민연금공단 이러닝시스템 사업의 총 예산은 얼마인가?",
  "ground_truth": "5억 4천만원",
  "source_doc": "RFP_국민연금공단_이러닝.pdf",
  "source_page": "12",
  "context_snippet": "본 사업의 총 예산은 금 540,000,000원이며...",
  "follow_up_id": null,
  "reviewer": "데이터B",
  "verified": true
}
```
